## T-027 ✅ Migration base módulo Calendario

Schema-only ticket que sienta tablas + RLS + audit (server actions vienen en T-028, UI en T-029). Migration `supabase/migrations/20260514125515_calendar_events.sql`.

**Tabla `public.calendar_events` (15 columnas)**: `id uuid pk default gen_random_uuid()` · `consultora_id uuid not null references consultoras(id) on delete cascade` · `tipo text not null check (tipo in (...))` con 7 valores `protocolo_anual|epp_entrega|capacitacion|calibracion|examen_medico|rgrl_anual|custom` · `titulo text not null check (length(trim(titulo)) between 3 and 200)` · `descripcion text check (descripcion is null or length(descripcion) <= 2000)` · `informe_id uuid references informes(id) on delete set null` · `fecha_vencimiento date not null` · `recurrence_months int check (recurrence_months is null or recurrence_months between 1 and 60)` · `status text not null default 'pending' check (status in ('pending','completed','cancelled'))` (sin `'snoozed'` por DA-04 — postergar es UPDATE de `fecha_vencimiento` real) · `completed_at timestamptz`+`completed_by uuid references auth.users(id) on delete set null` · `reminder_offsets_days int[] not null` (sin default SQL: T-028 server action inyecta el array correcto según tipo) · `metadata jsonb check (metadata is null or pg_column_size(metadata) <= 4096)` · `created_by uuid references auth.users(id) on delete set null` · `created_at`+`updated_at timestamptz not null default now()` con trigger `set_updated_at_calendar_events` que reusa `public.set_updated_at()` de T-011.

**2 indexes**: `idx_calevents_consultora_fecha` partial `WHERE status='pending'` sobre `(consultora_id, fecha_vencimiento)` para query principal del calendario (mantiene el índice chico, completed/cancelled fuera del caso de uso); `idx_calevents_informe` partial `WHERE informe_id IS NOT NULL` para lookup T-036 modal post-firma.

**Tabla `public.calendar_event_reminders` (8 columnas)**: `id uuid pk` · `event_id uuid not null references calendar_events(id) on delete cascade` · `consultora_id uuid not null references consultoras(id) on delete cascade` (denormalizado deliberadamente para RLS fast-path sin join al parent — mismo trade-off que T-024 `informe_attachments`) · `offset_days int not null check (offset_days >= 0 and offset_days <= 365)` · `scheduled_at timestamptz not null` · `status text not null default 'pending' check (status in ('pending','sent','skipped','failed'))` · `sent_at timestamptz` · `created_at` · `unique (event_id, offset_days)` (capa DB del idempotence stack del Discovery § 7.4 — previene dups si T-028 reintenta crear).

**2 indexes**: `idx_reminders_due` partial `WHERE status='pending'` sobre `scheduled_at` (índice principal del cron T-031: "reminders due cuyo status sigue pending") + `idx_reminders_event` sobre `event_id` (lookup "reminders de este event" para UPDATE de fecha → recalc).

**RLS calendar_events (3 policies)**: `calendar_events_select_own` USING `is_member_of_consultora(consultora_id)` (cualquier member de la consultora ve los eventos) · `calendar_events_insert_own` WITH CHECK `is_member_of_consultora(consultora_id) AND created_by = auth.uid()` (auto-atribuido, no usurpa identidad) · `calendar_events_update_own_or_owner` USING+WITH CHECK `is_member_of_consultora(consultora_id) AND (created_by = auth.uid() OR is_owner_of_consultora(consultora_id))` (creator OR owner). DELETE en events **default-deny para authenticated** (sin policy) — soft-delete UX = UPDATE status='cancelled'; hard-delete solo via service-role (cleanup admin / cascade desde consultoras).

**RLS calendar_event_reminders (1 policy)**: `calendar_event_reminders_select_own` USING `is_member_of_consultora(consultora_id)` (transparencia para el user de "qué voy a recibir y cuándo"). INSERT/UPDATE/DELETE en reminders **SIN policy para authenticated** = service-role only (decisión deliberada: si abriera a authenticated, un user podría programar `scheduled_at` arbitrario o duplicar reminders bypassing la UNIQUE constraint). T-028 server actions crean reminders via `createServiceRoleClient()`; T-031 cron marcará `sent/skipped/failed` también via service-role. Helpers SQL T-015 (`is_member_of_consultora`, `is_owner_of_consultora`) reutilizados — **sin nuevas funciones SQL en T-027**.

**Audit trigger `audit_calendar_events()`** `security definer set search_path = ''` (defensa contra search_path hijacking) + 3 triggers AFTER INSERT/UPDATE/DELETE escriben a `audit_log` con `actor_user_id = auth.uid()`. **Diff guard del UPDATE**: row de audit solo se escribe si `(titulo, tipo, status, fecha_vencimiento, recurrence_months, descripcion, completed_at) is distinct from (old.*)` — `metadata` y `reminder_offsets_days` deliberadamente excluidos del guard (cambios en ellos NO disparan audit row porque no son business-meaningful: metadata es payload para futuros campos custom + offsets son recomputables). `descripcion` está en el guard pero NO en el `jsonb_build_object` de `before_data`/`after_data` payload (puede pesar hasta 2 KB y satura el row de audit_log). Payload INSERT incluye `tipo,titulo,status,fecha_vencimiento,recurrence_months,informe_id`; UPDATE incluye `titulo,tipo,status,fecha_vencimiento,recurrence_months,completed_at`; DELETE incluye `titulo,tipo,status,fecha_vencimiento`. **NO hay audit trigger sobre `calendar_event_reminders`** (deliberado: operacional, no de negocio — el lifecycle sent/skipped/failed vive en su propia columna `status` + `sent_at`). Helpers T-015 reusados + sin nuevas funciones SQL en T-027.

FKs a `clientes`/`empleados` omitidas porque esas tablas no existen aún (la migration que las cree agregará las columnas + FK + indexes).

**4 server actions con discriminated union return** (patrón canónico T-019/T-024 — NUNCA throw, siempre return `{ok:true,...}|{ok:false,code,message,fieldErrors?}`):

- `createCalendarEventAction(input)` — Zod parse → getUser → getCurrentConsultora → resolve offsets (input o `DEFAULT_REMINDER_OFFSETS_BY_TYPE[tipo]`) → INSERT event con cliente authed (RLS WITH CHECK valida member + auto-attrib `created_by=auth.uid()`) → si Postgres devuelve code `42501` (RLS rejection cross-tenant o spoof) mapea a `FORBIDDEN` (alineado con T-024 attachment pattern) → bulk INSERT reminders via `createServiceRoleClient()` (RLS bloquea INSERT a authenticated en reminders, decidido T-027) → **rollback manual del event** si bulk INSERT falla post-event-INSERT (no hay transactions cross-client en Supabase, mismo patrón que T-024 storage) → `revalidatePath('/calendario')`+`/dashboard` → return `{ok:true, eventId, remindersCreated, remindersSkippedPast}`.

- `updateCalendarEventAction(eventId, patch)` — Zod (eventId UUID + patch refine "al menos un campo") → SELECT event vía cliente authed (RLS filtra cross-tenant → null=NOT_FOUND) → permission gate defensivo `isCreator || isOwner` (FORBIDDEN si false) → UPDATE event authed con `.select('id')` post-op para detectar race RLS WITH CHECK → **3 casos de recálculo reminders**: (A) patch trae `reminder_offsets_days` → DELETE reminders pending vía service-role + INSERT nuevos vía `computeReminderRows` (sent/failed previos se preservan); (B) patch trae solo `fecha_vencimiento` (sin offsets) → loop por reminders pending, recomputar `scheduled_at` con `computeScheduledAtUtc(newFecha, r.offset_days)`, UPDATE inline preservando `id+offset_days` (UNIQUE constraint intacta), si el nuevo `scheduled_at < now` → marcar `status='skipped'`; (C) patch no toca fecha ni offsets → no tocar reminders. Return `{ok:true, eventId, remindersRecomputed, remindersCreated, remindersSkippedPast}`.

- `completeCalendarEventAction(eventId)` — Zod UUID + auth + consultora + SELECT event → check `status === 'pending'` (sino `ALREADY_FINAL`) → permission gate creator OR owner → UPDATE `status='completed', completed_at=now(), completed_by=auth.uid()` → UPDATE pending reminders → `skipped` vía service-role (no spam post-complete) → **auto-recurrencia Opción A**: si `event.recurrence_months !== null` → INSERT next event con `addRecurrenceMonths(fecha, N)` + mismo `tipo,titulo,descripcion,recurrence_months,reminder_offsets_days,metadata` pero **`informe_id=null` en next** (decisión: el informe original es el doc del periodo actual, no el siguiente — DA-05 modal post-firma re-vincula con el informe N+1) + `created_by=auth.uid()` del completador (no preserva original) + nuevos reminders vía `computeReminderRows`. **Best-effort: si la auto-recurrencia falla NO revierte el complete** del original — el user vino a marcar "lo hice", no a "creá el siguiente"; errores van a Sentry vía `logger.error` con contexto suficiente (`originalEventId, fechaVencimiento, recurrence_months, err`) para recovery manual admin. Return `{ok:true, eventId, nextEventId, remindersSkipped, nextRemindersCreated, nextRemindersSkippedPast}`.

- `cancelCalendarEventAction(eventId, reason?)` — análogo a complete pero `status='cancelled'` + opcional reason en `metadata.cancel_reason` (merged con metadata previo) + UPDATE pending reminders → skipped + **NO crea next aunque haya `recurrence_months`** (cancelar != completar; anti-test cubierto). `cancel_reason` queda en metadata pero NO en audit_log (excluido del diff guard T-027), aceptable porque `status='cancelled'` ya queda en audit.

**NO `deleteCalendarEventAction`** — RLS default-deny para authenticated por T-027 (soft-delete vía cancel).

**5 queries de lectura** en `queries.ts` con `import 'server-only'` defensivo (sin `'use server'` para que sean importables desde Server Components + Server Actions): `getCalendarEventsForConsultora(supabase, options?)` paginado con filtros `status[]`+`tipo[]`+`fechaFrom`+`fechaTo`+`limit`+`offset`, ORDER BY fecha ASC + id; `getCalendarEventById(supabase, eventId)` devuelve `{event, reminders[]}` (2 queries separadas en lugar de embedded join para shape predecible + typed); `getUpcomingEvents(supabase, daysAhead=30)` para panel "próximos vencimientos" dashboard T-030 (BETWEEN today AND today+N); `getOverdueEvents(supabase)` status='pending' AND fecha < today; `getEventsByInformeId(supabase, informeId)` para T-036 modal post-firma (detectar si el informe ya tiene vencimiento auto-creado). **Todas reciben client authed** como primer arg → RLS automático filtra por consultora del JWT; NO toman `consultora_id` como param (fuente de verdad es el claim, no el caller).

**Helper compartido `src/shared/calendar/scheduling.ts`** reusable por T-031 cron: `computeScheduledAtUtc(fechaIso, offsetDays)` construye `Date.UTC(year, month-1, day-offset, SCHEDULED_AT_SEND_HOUR_UTC=12, 0, 0)` evitando sorpresas del TZ del runtime (VPS Hostinger) — Argentina UTC-3 fijo sin DST desde 2009 → 09:00 ART = 12:00 UTC; setting per-consultora llega cuando entren tenants Chile/Uruguay Fase 5 → follow-up T-028-FU3. `computeReminderRows({eventId, consultoraId, fechaVencimientoIso, offsetDays[], now})` devuelve `{rows[], skippedPast: number}` — para cada offset construye row con `scheduled_at`, si `scheduledAt < now` skipea silencioso (NO persiste fila `status='skipped'` para mantener UNIQUE limpia para futuros UPDATE de fecha que reincorporen el offset; Discovery § 5.5). `addRecurrenceMonths(fechaIso, months)` wrapper de `date-fns/addMonths` (handle correcto fin-de-mes: Jan 31 + 1 = Feb 28/29 bisiesto, Mar 31 + 1 = Apr 30 — NO Mar 3 ni May 1 como Date nativo).
