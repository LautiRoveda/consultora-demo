## T-047 ✅ Migration tabla clientes

Sprint 4, primer ticket módulo Clientes — schema-only, sienta el patrón forward para Empleados/EPP/Pagos/Checklists que vienen después. PR `#84` merge SHA `0e2c3a4`. Cierra el problema del form RGRL (T-021) que hoy pide `razon_social`/`cuit`/`domicilio`/`localidad`/`provincia` como texto libre cada vez que el consultor genera un informe — con T-047..T-051 el consultor da de alta sus clientes una vez y los selecciona del dropdown al generar informe (autocompletando los fields).

**Campos de la tabla** (15 business): `razon_social text not null check (length between 2 and 200)` · `cuit text not null check (cuit ~ '^\d{2}-\d{8}-\d{1}$')` (formato normalizado XX-XXXXXXXX-X matcheando output de `normalizeCuit()` de `src/shared/templates/common/cuit.ts`) · `nombre_fantasia text` opt (PYMEs usan "El Galpón" en lugar de razón social formal) · `domicilio text` opt · `localidad text` opt · `provincia text` opt (texto libre en SQL, UI hace dropdown del enum `PROVINCIAS_AR` de `common/site.ts` para forzar valor válido; CHECK estricto NO porque queremos abrir a tenants chilenos/uruguayos Fase 5 sin migration) · `contacto_nombre text` opt · `contacto_email text` opt (regex permisivo `^[^@\s]+@[^@\s]+\.[^@\s]+$`, Zod hace validación real en T-048) · `contacto_telefono text` opt · `industria text` opt (texto libre por ahora, no enum SQL — categoría puede crecer cuando crucemos data real con amigos HyS) · `art text` opt · `notas text check (length <= 2000)` opt.

**Indexes**: `idx_clientes_consultora_razon_social` partial sobre `(consultora_id, razon_social)` para list query principal con sort alfa + `idx_clientes_consultora_cuit` partial UNIQUE sobre `(consultora_id, cuit)` (este SI es el UNIQUE constraint del punto siguiente, sin duplicar) — **UNIQUE (consultora_id, cuit) WHERE archived_at IS NULL**: dos clientes activos del mismo tenant con el mismo CUIT es error de data entry, pero archivar uno y dar de alta otro con mismo CUIT (cambio razón social, fusión) DEBE permitirse; cross-tenant SI permitido (dos consultoras distintas tienen el mismo cliente físico).

**RLS** (3 policies + DELETE default-deny): `clientes_select_own` USING `is_member_of_consultora(consultora_id)` (cualquier member ve todos los clientes del tenant — Diego con 5 técnicos, coordinación de equipo); `clientes_insert_own` WITH CHECK `is_member_of_consultora(consultora_id) AND created_by = auth.uid()` (auto-atribución, no usurpa identidad); `clientes_update_own` USING + WITH CHECK `is_member_of_consultora(consultora_id)` (any member edita — **DIFERENCIA con `informes` que tiene gate creator OR owner**: clientes son data compartida del tenant, no borradores personales del consultor; cualquier técnico que visita puede updatear teléfono/contacto del cliente; si más adelante hay fricción real, hardenear a owner-only); DELETE sin policy para authenticated (default-deny, soft-delete UX via `UPDATE archived_at = now()`, hard-delete admin-only via service-role).

**Audit trigger** `audit_clientes()` `security definer set search_path = ''` replicando patrón T-027 con variables `declare` (patrón T-024): INSERT payload (6 fields: `razon_social`/`cuit`/`nombre_fantasia`/`industria`/`localidad`/`provincia`); UPDATE diff guard sobre **12 fields** mutables (`razon_social, cuit, nombre_fantasia, domicilio, localidad, provincia, contacto_nombre, contacto_email, contacto_telefono, industria, art, archived_at`) — `notas` excluido del guard Y del payload (puede pesar hasta 2 KB, satura audit_log; **matchea T-027 patrón `descripcion` excluido de ambos**); UPDATE payload before/after con los 12 fields del guard; DELETE payload (4 fields: `razon_social`/`cuit`/`nombre_fantasia`/`archived_at`); actions canónicos `'created'`/`'updated'`/`'deleted'` (verbos T-019/T-024/T-027).

**Cascade**: `consultora_id REFERENCES consultoras(id) ON DELETE CASCADE` borra clientes al borrar consultora; `created_by ON DELETE SET NULL` preserva clientes huérfanos sin actor cuando borrás el user.
